## 一、问题与现状

### 白嫖路径（已核实）

外部拿到 `~/.mclaw/openclaw.json` 里 `models.providers.<id>.apiKey` 的 **runtime key**（UUID v4）+ 公网地址，即可绕过 mclaw 客户端：

```bash
curl https://[REDACTED]/v1/chat/completions \
  -H "Authorization: Bearer <runtime-key-uuid>" \
  -H "Content-Type: application/json" \
  -d '{"model":"<绑定的模型名>","messages":[{"role":"user","content":"hi"}]}'
```

### 现状缺口

| 项 | 现状 |
|----|------|
| `/v1` 路由中间件 | **无 auth 中间件**，唯一门是 `extractToken`+`resolveModel` 只查 key 存在性（`proxy.go:128-186`） |
| runtime key 时效 | **无过期、无撤销**，UUID 永久有效（`ent/schema/modelapikey.go` 无 expires/revoked 字段） |
| 设备/客户端绑定 | **无**，只绑 user_id+model_id |
| 计费阻断 | **事后扣费，失败只 Warn 不阻断**（`proxy.go:330-335`） |
| runtime key 存储 | **明文**存 openclaw.json + electron-store（非 OS keychain） |

### 两个 token 的区别（别搞混）

| | gateway token（`clawx-...`/`mclaw-...`） | runtime key（UUID） |
|---|---|---|
| 用途 | 桌面端 ↔ **本地 loopback** Gateway | Gateway 代用户调**公网** llmproxy |
| 暴露面 | 仅 127.0.0.1，外网不可达 | **公网可达** |
| 能白嫖 | 单独不能 | **能，这是钥匙** |

## 二、第一期方案：客户端签名 + runtime key 时效

> 目标：堵住白嫖。不上全套（无限流/设备绑定/事前额度门，留第二期 YAGNI）。

### 第一刀：客户端 HMAC 签名（绑定的核心）

**原理**：只有 mclaw 客户端能算出签名（因为只有它有 deviceSecret），纯 curl 拿着 runtime key 也算不出，llmproxy 验签不过直接 401。这是"绑定 mclaw 客户端"的本质。

#### 2.1.1 deviceSecret 的生成与存储

- 桌面端首次启动（或登录后）生成一个 `deviceSecret = crypto.randomBytes(32).toString('hex')`，**64 字符**。
- 存储：**macOS Keychain**（Electron `safeStorage.encryptString`），**不落 openclaw.json**。
  - 现 secret-store 是明文 electron-store（`electron/services/secrets/secret-store.ts`），本期新增一个 keychain 后端专存 deviceSecret。
  - Windows 用 DPAPI（safeStorage 自动适配），Linux 用 secret-service。
- deviceSecret 不发给后端明文；后端只存它的**哈希摘要**用于验签（见 2.1.3）。

#### 2.1.2 签名算法

每个发往 `/v1/*` 的请求带签名头：

```
X-Mclaw-Sig: t=<unix秒>, v1=<hex>
```

其中 `v1 = HMAC-SHA256(deviceSecret, "<t>\n<method>\n<path>\n<bodySha256>")`

- `t`：请求时间戳（秒），llmproxy 校验 ±5 分钟内，防重放。
- `method`：大写 HTTP method（POST）。
- `path`：请求路径（如 `/v1/chat/completions`）。
- `bodySha256`：请求体的 SHA-256 hex（防篡改）。

#### 2.1.3 后端验签（llmproxy）

- `modelapikey` schema 新增 `device_secret_hash`（string，存 deviceSecret 的 SHA-256 摘要）。
- 桌面端 `IssueRuntimeKey` 时把 deviceSecret 的哈希一起提交给后端，后端存到该 key 记录。
- `proxy.go` 新增验签中间件（或在 ServeHTTP 里）：
  1. 取 `X-Mclaw-Sig` 头，解析 `t` 和 `v1`。
  2. 校验 `t` 在 ±5 分钟窗口内。
  3. `resolveModel` 拿到 key 记录 → 取 `device_secret_hash`。
  4. 重算 HMAC（用 deviceSecret 哈希不行——HMAC 要原 secret）。

  **⚠️ 这里有个坑**：HMAC 验签必须用原 deviceSecret，但后端只存哈希就验不了。两个解法（老王推荐 A）：

  - **方案 A（推荐）**：deviceSecret 本身作为对称密钥，后端**明文存** deviceSecret（或加密存）。验签时取出明文重算 HMAC 对比。安全性靠"deviceSecret 不在 openclaw.json、只进 keychain + 后端 DB"保证。泄露面 = 后端 DB 被脱裤，可接受（DB 本来就存着更强的 runtime key）。
  - 方案 B：改用非对称签名（客户端持私钥存 keychain，后端存公钥验签）。更安全但复杂，第二期考虑。

  → **第一期用方案 A**：后端 `modelapikey` 存 `device_secret`（明文），验签重算 HMAC。KISS。

  5. 对比 `v1`，不等则 401。
  6. 校验 `bodySha256` == SHA256(实际 body)，不等则 401（防篡改）。

### 第二刀：runtime key 时效 + 撤销

#### 2.2.1 schema 变更

`ent/schema/modelapikey.go` 新增：
- `expires_at`（Time, Optional）：默认签发时 `now + 24h`。
- `revoked_at`（Time, Optional）：撤销时间，非空即失效。

#### 2.2.2 签发逻辑（IssueRuntimeKey）

- `CreateRuntimeAPIKey` 签发时设 `expires_at = now + 24h`。
- 复用已有 key 时（`GetRuntimeAPIKeyByUserModel`）：若已过期或快过期（< 1h），签发新 key 替换。
- 返回新 key 给客户端，客户端更新 provider account 的 apiKey。

#### 2.2.3 续签机制（桌面端）

- 桌面端启动时 + 每 1 小时检查当前 runtime key 是否快过期（这个"过期时间"后端随 key 一起返回）。
- 快过期则调 `IssueRuntimeKey` 续签，更新本地 provider account 的 apiKey + 过期时间。

#### 2.2.4 resolveModel 校验

`proxy.go:resolveModel` 拿到 key 后：
- `revoked_at` 非空 → 401。
- `expires_at` 早于 now → 401。

## 四、风险与边界

- **签名注入点**：若 OpenClaw Gateway 出站请求无法 hook 加自定义头，则签名必须在桌面端层面注入——但实际 HTTP 是 Gateway 发的，可能需要改 Gateway 配置或用本地代理拦截。**这是最大技术风险**，任务 2.2 需先验证可行性。
- **老 key 兼容**：migration 给现有 key 填 24h 过期 + 空 device_secret（验签跳过？不行，会留后门）。**决策**：老 key 设 `device_secret` 为空 → 验签时若 device_secret 为空则**强制要求签名**（即老 key 必须重新签发才能用），或给老 key 一个宽限期。倾向：migration 后老 key 直接过期，客户端首次启动自动续签带 deviceSecret。
- **deviceSecret 明文存后端**：方案 A 的取舍。后端 DB 被脱裤才泄露，且 DB 本就存 runtime key，风险等级不变。
- **时间漂移**：±5 分钟窗口，客户端时钟严重偏差会误拒。可放宽到 ±10 分钟。
- **非对称签名**：第二期可升级（私钥 keychain + 公钥后端），彻底消除 deviceSecret 明文落库。
